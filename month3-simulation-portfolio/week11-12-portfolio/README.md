### Day 1-3: メインプロジェクトの仕上げ

#### Day 1: コードの棚卸しとクリーンアップ

```
コード品質向上の作業リスト:

  1. 不要コードの削除
     ├── デバッグ用のprint文
     ├── コメントアウトされた古いコード
     ├── 未使用のimport文
     └── テスト用の一時ファイル

  2. コードの整理
     ├── 関数名・変数名を意味のある名前に統一
     ├── docstringの追加（関数、クラスに日本語/英語で）
     ├── 型ヒントの追加（Python 3.10+ の型アノテーション）
     ├── 定数の定義（マジックナンバーの排除）
     └── ファイル構成の整理（1ファイルが長すぎないか）

  3. スタイルの統一
     ├── フォーマッター適用（black, isort）
     ├── リンター実行（ruff）
     └── 命名規則の統一（snake_case for Python）

  4. 設定の外部化
     ├── ハードコードされたパスを設定ファイルへ
     ├── APIキー等の秘密情報を.envファイルへ（.gitignoreに追加）
     └── .gitignoreの整備（.pyc、__pycache__、models/等）

  5. 依存管理
     ├── requirements.txt の整備（pip freeze → 手動整理）
     ├── pyproject.toml の作成（推奨）
     └── Python / ROS 2 バージョン明記
```

**ファームウェアエンジニアとしての強み**:
コードレビュー、設計文書作成、品質管理の経験が直接活きる。
組込み開発で培った「仕様書に基づくコード品質」の意識をポートフォリオコードにも適用する。

### Day 11-12: ボーナスプロジェクト & スキル確認

#### Day 11: KiCadカスタムセンサーボード設計（ボーナス）

ファームウェアエンジニアとしての**唯一無二の強み**を見せるプロジェクト。

**センサーボードコンセプト:**

| 部品 | 型番 | 機能 |
|------|------|------|
| MCU | ESP32-S3 | Wi-Fi/BLE、micro-ROS対応 |
| ToFセンサー | VL53L5CX | マルチゾーン距離測定 |
| 環境センサー | BME680 | 温湿度気圧 |
| IMU | ICM-42688-P | 6軸慣性計測 |
| マイク | SPH0645LM4H | I2S MEMSマイク |

基板サイズ: 50mm x 30mm、電源: 5V（USB-C）、通信: USB-C + Wi-Fi（ROS 2 DDS）

**KiCad設計の流れ:**
1. 回路図設計（ESP32-S3基本回路、センサーI2C/SPI/I2Sバス、電源回路、USB-C）
2. PCBレイアウト（GNDプレーン、デカップリング配置、シルク印刷）
3. BOM（部品表）作成
4. DRC/ERCチェック、3Dビュー確認
5. 設計データ一式をGitHubに公開

**技術的な差別化ポイント**:
```
KiCadで回路設計からPCBレイアウトまで行える。
micro-ROS対応にすることで、このボードから直接
ROS 2のトピックにセンサーデータをパブリッシュできる。

→ Physical AI 領域において、
   HW/SW両方をカバーできるスキルは大きな差別化要因
```

## つまずきやすいポイントと対処法

### 1. 完璧主義の罠

**対処法**: "Done is better than perfect"。動くデモ > 完璧なコード。
タイムボックスを設定（コードクリーンアップ: 最大1日、デモ動画: 最大2日、README: 最大1日、
ブログ1本: 最大半日）。80%の完成度で一旦リリースし、後から改善する。

### 2. デモ録画の問題

**対処法**: OBSのエンコーダをNVENC（GPU）に設定してCPU負荷軽減。
デモのコマンドは事前にスクリプト化してコピペ。複数テイク撮影して良いものを選ぶ。
最悪、スクリーンショットのスライドショーでもOK。

### 3. GitHubのREADMEフォーマット

**対処法**: 画像はリポジトリ内の`docs/images/`に配置し相対パスで参照。
Mermaid図は複雑すぎるなら画像化。VS CodeのMermaid拡張でローカルプレビュー。

### 4. ブログ執筆に時間がかかりすぎる

**対処法**: アウトラインを先に書き肉付け。コードは既存コードを流用（新規は書かない）。
推敲は1回まで。「60%で公開、反応見て改善」の精神。

### 5. 統合時の問題

**対処法**: 依存パッケージのバージョン競合 → 仮想環境を新規作成。
GPU メモリ競合 → AI推論とシミュレーションの同時実行を避ける。
全体が動かなくても各コンポーネント単体のデモは残す。

### 6. スコープクリープ

**対処法**: Must / Should / Could / Won't で優先順位を明確化。

| Must | Should | Could | Won't（今回はやらない） |
